    Distributed Rendering Structure

    Network Connection General Rules:

    https://stackoverflow.com/questions/7664109/broadcasting-ipport-by-socket-server

    • Servers usually listen to requests and send replies
    • A server that sends regular “announce” broadcast packets over a WiFi network, for a client that may arrive or not, wastes the network bandwidth, while a client knows exactly when it needs to poll for available servers, and stop once it’s done.

    Examples

    Unreal Swarm Farm:

    • Swarm agent: client, initialize connection by pinging coordinator by hostname/ip (which is pre-configured)
      • the agent can send lightmap bake requests to the server, and also can process lightmap bake jobs from the server
    • Swarm coordinator: server
      • Observation: the coordinator gets bake requests immediately (which makes sense because it is a request from a client), but also assigns the job to agents (how does a server send a job without a request? does it broadcast? or does the agent request for job periodically?)

    Blender Network Rendering (Legacy):

    • Client: client, only sends render requests
      • Observation: client can request to see connected slaves after connecting to the master, which makes sense; it can send requests to master
    • Master: server, coordinate (process requests from clients and assign jobs to slaves)
      • Observation: master broadcasts address? what does this mean? looking for slaves to assign jobs to?
      • it can get requests immediately (which makes sense), same question as before, how does it assign the job to a slave?
    • Slave: a different client, only used to process render job
      • Observation: needs a pre-configured address to connect to the master and waits for jobs

    How does the server/master/coordinator/manager send job to the slave/agent/render node

    1. broadcast
    2. the client send request periodically
    3. a listener server on client side that the server pings to
